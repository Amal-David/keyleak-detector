Confirmed Findings
Critical Open Table: payment_details confirmed
Financial payment data readable without authentication. Anyone with the anon key can query earnings, payout methods, and account details.
GET /rest/v1/payment_details?select=*&limit=1 → 200 OK — columns: id, user_id, payout_method, account_details, created_at
Fix: ALTER TABLE payment_details ENABLE ROW LEVEL SECURITY; then create policies restricting access to the owning user.
High Open Table: profiles confirmed
User profile data including usernames, emails, and account settings returned to unauthenticated callers.
GET /rest/v1/profiles?select=*&limit=1 → 200 OK — columns: id, username, display_name, avatar_url, email, settings
Fix: ALTER TABLE profiles ENABLE ROW LEVEL SECURITY; — allow public reads only on non-sensitive columns.
High Open Table: orders confirmed
Full order history readable. select(*) returns all columns including internal metadata and pricing.
GET /rest/v1/orders?select=*&limit=1 → 200 OK — columns: id, user_id, total_cents, status, created_at
Fix: Enable RLS. Restrict select() to the owning user — never expose other users' order data.
High Public Storage Bucket: uploads confirmed
The "uploads" storage bucket is marked public. Anyone can list and download all uploaded files.
GET /storage/v1/bucket → 200 OK — {name: "uploads", public: true}
Fix: Set bucket to private in the Supabase dashboard. Add storage policies for authorized upload/download only.
Medium Unauthenticated RPC: increment_views confirmed
The increment_views function is callable with just the anon key. An attacker can inflate view counts for any record.
POST /rest/v1/rpc/increment_views → 200 OK
Fix: Add auth.uid() check inside the function or revoke EXECUTE from the anon role. Add rate limiting.
Low Wide-Open CORS confirmed
REST API returns Access-Control-Allow-Origin: *. Any origin can make authenticated requests.
access-control-allow-origin: *
Fix: Restrict CORS to your application's origin in the Supabase dashboard.

Additional Leads
High Client-Side Admin Check
Admin authorization is a browser-side boolean. Admin operations (manage users, process refunds) are gated only by a JS check that can be bypassed in the console.
if (user.is_admin === true) { showAdminPanel(); }
Fix: Move admin checks to RLS policies or Postgres functions. Never trust isAdmin from the client.
Low select("*") Overfetch — 9 tables
9 of 12 client-side queries use select("*"), fetching every column including internal fields. Even with RLS, this leaks column names and potentially sensitive data.
Fix: Replace .select("*") with explicit column lists for every client query.
