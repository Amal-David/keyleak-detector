RUNTIME SECURITY

The Key Your Scanner Swore Wasn't There

GitLeaks, TruffleHog, and GitHub secret scanning all came back green — and a researcher still pulled a live Stripe key out of your running app in under a minute.

May 29, 2026 · 7 min read

The team did everything right. GitLeaks on every commit. TruffleHog in the CI pipeline. GitHub secret scanning watching the repo. Every push came back green, every pull request clean. Then a researcher opened the production site, popped open dev tools, and read a single minified JavaScript file. Inside it, in plaintext, sat a live Stripe sk_live_ key — full API access, real money. They had it in under a minute.

The secret was never in the repo. Not in a commit, not in git history, not in any file a source scanner could read. It was injected at build time and shipped to every visitor. The lesson is uncomfortable and simple: your source code can be spotless while your running app bleeds.

The structural blind spot

This is not a tuning problem. You cannot regex your way out of it. GitLeaks, TruffleHog in repo mode, and GitHub secret scanning all operate on source files and git history. They are very good at that. But a whole class of secrets only comes into existence after the build: environment variables inlined into bundles, runtime config fetched when the app boots, values returned in API response bodies and headers, and original variable names and routes re-exposed by source maps.

The artifact a source scanner would need to inspect — the minified bundle, the hydrated config object — does not exist in the repository. It is manufactured at build time and lives only in the deployed app. Intruder's research frames the gap bluntly: SAST tools "don't cover the full picture; secrets within JavaScript bundles slip through gaps," and infrastructure scanners "ignore JavaScript files that browsers would load." You are asking a tool to find something in a place it was never written.

How secrets reach the runtime but never the repo

Three modern delivery paths put secrets in front of users while keeping the repository clean.

Build-time inlining. Next.js (NEXT_PUBLIC_*) and Vite (VITE_*) replace env-var references with their literal string values at build and bake them into the client bundle. Give a server secret the wrong prefix — NEXT_PUBLIC_STRIPE_SECRET=sk_live_... — and the literal key is hardcoded into shipped JavaScript. The repo only ever held process.env.NEXT_PUBLIC_STRIPE_SECRET, a perfectly innocent-looking reference. The scanner finds nothing because there is nothing to find. Vercel now shows dashboard warnings precisely because this footgun is so common.

Edge and serverless config injection. On Vercel, Netlify, and Cloudflare, env vars live in the platform dashboard and are hydrated at deploy — never committed. That is good hygiene for server secrets. But the moment one reaches the client through inlining or a leaky public config endpoint, the only place to catch it is the running deployment, because it was never in a file git ever saw.

BaaS clients that ship keys by design. Supabase anon keys and Firebase web configs are meant to be in the browser. More on why that is its own trap below.

Source maps make all of this worse. When shipped to production by accident — extremely common — they reverse minification, restoring original variable names, file paths, and inlined constants, handing an attacker a readable map of your internal API routes. Source maps live in the deployed artifact, not the source tree a commit scanner reads.

Public-by-design is not safe-by-default

Here is where intuition fails. A Supabase anon key is supposed to be public. The security boundary was never the key — it's the server-side Row Level Security policy behind it. A static scanner sees a string and has two options: ignore it (correctly, it's "public") or flag it without any way to reason about whether the backend is actually locked down.

And the default is exposed. Supabase auto-generates a REST API from your Postgres schema, but Row Level Security is off by default on tables created via SQL or the Table Editor. The app works flawlessly while unauthenticated requests carrying the public anon key quietly read and write protected rows. You cannot see this in source code. The only way to know is to issue a live, unauthenticated request and watch protected data come back. That question — does this public key return data it shouldn't? — is the dimension static tools cannot supply.

Exploitability is the real signal

A pattern match is a guess. A live probe is proof. The right question is not "does this look like a key?" but "is this key live, and what can it touch right now?"

This is not theoretical. TruffleHog built its reputation on verifying detectors that authenticate against the real service and classify findings as verified, unverified, or revoked. Runtime scanning extends that idea to the deployed app itself: extract the secret and prove it works, probe whether a public BaaS key returns protected rows, and tell an active threat apart from a dead string. That is the difference between a noisy candidate list and a ranked incident queue.

And dead strings are rarer than anyone hopes. GitGuardian retested credentials it had confirmed valid in 2022 and found the validity rate was still above 64% in January 2026 — with 70% of 2022-leaked secrets still active two years on. Leaked keys are almost never rotated. Exploitability is not a corner case; it is the norm.

The gap is widening, not closing

Three trends push more secrets into the runtime surface, and adoption only accelerates them. SPAs and client-heavy frameworks move more logic and config into the browser. Edge and serverless platforms normalize injecting config outside git. And AI app generators and BaaS-first scaffolding ship front-end-first apps with security defaults off. Every one of these relocates secrets and trust boundaries from the repo — where static tools look — to the running app, where they can't. The problem does not self-correct.

The leak volume confirms the direction of travel. GitGuardian recorded 28.65 million new hardcoded secrets in public GitHub commits in 2025 — a 34% year-over-year jump — and AI-service secrets alone hit 1,275,105 detections, up 81%. That is just what was visible in public commits. The runtime surface is the part nobody is counting.

Truffle Security scanned 400TB of web data and found 11,908 live secrets across 2.76 million pages — including roughly 1,500 Mailchimp API keys hardcoded directly into front-end HTML and JavaScript.

The receipts

This is not a hypothetical risk pattern. It is being exploited at scale, and the evidence is everywhere you point a runtime probe.

  • Intruder remotely scanned around 5 million applications and surfaced 42,000+ exposed tokens across 334 secret types — live GitHub and GitLab tokens, active Slack webhooks, and more, all living in JavaScript bundles that SAST and infra scanners structurally miss.
  • Cremit's study of Vercel-hosted services found a small but lethal slice shipping valid production secrets — Stripe sk_live_ keys, GitHub tokens, AWS secret access keys — straight into client bundles via NEXT_PUBLIC_ misuse.
  • CVE-2025-48757: researchers found Supabase tables across many Lovable-generated apps that were readable by unauthenticated requests using the public anon key — missing RLS, provable only at runtime.
  • Misconfigured Firebase has exposed enormous troves of user records and plaintext passwords across thousands of sites — a pure BaaS-rules failure invisible to source scanning.
  • Cyble found roughly 3,000 live production websites exposing active OpenAI API keys directly in client-side JavaScript.
  • The Rabbit R1 leaked hardcoded API keys that stayed valid well after notification; a DOGE developer's xAI key, found externally by GitGuardian, remained live even after public disclosure. The detection happened — too late, and the key kept working.

For scale on why "too late" is the default outcome: in a controlled honeypot, an AWS key posted to a public repo was discovered and abused within one minute of exposure.

Complement, not replacement

Be clear about the framing, because the security audience will reject a false binary. Source scanners catch secrets before they ship and protect git history — that is the right place to stop a secret early, and you should keep every one of them. Runtime scanning catches what survives the build and reaches users, and proves exploitability. "Use this instead of GitLeaks" is wrong. The mature posture runs both: shift-left prevention on commits, plus deploy-time verification of the actual artifact users receive. Runtime scanning is the missing right-hand bookend of the pipeline.

Make it a launch gate

The operational home for this is a CI gate against your preview or staging deploy, after build but before promote-to-prod. Fetch and inspect the real built bundle. Hit the live endpoints the way an attacker would. Validate any key you find. Probe BaaS tables for missing RLS. Then fail the gate on a verified-live, high-severity finding — per PR, on the exact artifact your users will get. A commit scan can never inspect that artifact, because it does not exist yet at commit time. This gate catches the inlining mistake and the RLS regression on the build that ships.

One objection blocks adoption: security teams will not send their bundles and keys to a third-party cloud. So don't. A runtime scanner that runs 100% locally — browser extension, CLI, and self-hosted Action — does live validation against the target's own endpoints without exfiltrating anything. That is what makes a deploy-time gate palatable.

Scan what attackers scan

The one line to keep: attackers don't scan your repo. They scan your running app. So scan it first — and scan it the way they would, against the live deployment, proving which keys are actually exploitable.
